node爬虫--豆瓣最热门的500部日本动画

发表于 2016-10-20   |   分类于 爬虫

  • 数据及源码：具体数据，封面图片，源码

  • 技术栈及库：

  • node
  • superagent，发送GET请求
  • cheerio，类jQuery，分析页面
  • async库，并发控制（不限制并发数量，立马被403，别问我为什么知道）

  • co，拒绝回调函数，yield控制流程

  • 过程：

  1. 获得动画网址的列表animelist.js。 分析豆瓣热门动画页，分析Network发现某个个GET请求，改变最后的数字直到500，可以获得这个数据_data/id-list-json。
  2. 获得具体动画数据app.js，先根据id-list-json发送superagent请求，用cheerio分析页面获得数据，其中图片pipe一个新的图片网址，具体在_images/id中，几个影评的网址发送几个新的superagent请求获得具体的影评，最后将数据存放在_data/id中，所有流程用co控制。
  3. 分析改造数据change-data.js到data/id，合并简要数据data-all.js到data/moives.json。

  • 感想与总结：

  1. 本爬虫时间2016-10-20，以后可以url和dom结构变动导致无法复用。
  2. 多分析Network，多node app 分析数据，问题总会解决。
  3. 第一次用10个并发+0.1秒延迟，途中就被403了，后来改用3个并发+5秒延迟，大概20分钟爬完。

demo——redux实现简单todos

发表于 2016-10-16   |   分类于 demo

  • demo地址：redux-todos

  • 源码：github源码

  • 技术栈：react，redux，babel，webpack

  • 特点：

  1. react + redux 实现简单的todos，ES6用babel转译。
  2. 数据存储在localStorage。
  3. 在之前做的react-todos基础上，用redux重写状态和事件

  • 感想与总结：

  1. redux相比vue的vuex真的真的好复杂~~，看了好久才大概明白。
  2. redux和vuex的一个显著区别：redux全部state和action由props在根组件向下传，vuex则在任何组件可以直接获得action，get state。
  3. reducers中每个state一个文件，最后汇总在index.js里。
  4. reducers如果是array，需要每一个case需要return […array]，如果直接return array将不会更新props。

  • 结构：

|   |   └── App.js ........ 包裹层与redux连接

发表于 2016-10-06   |   分类于 demo

  • 源码：github源码

  • 技术栈：

  1. 框架：vue，vuex, vue-router
  2. 样式：sass
  3. 图表UI：ichart

  5. 打包工具：webpack

  • 特点：

  1. 集成现在个人所会的技术栈，具体构造见最后的结构图。
  2. vue-router控制路由，共有问卷列表，填写问卷，问卷详情，新建问卷，4个路由页面。
  3. vuex控制数据流向，有两个具体修改数据的事件，创建新建问卷和提交答卷。数据存储在localStorage。

  4. 最后踩了个坑，JSON.stringify()会将Date转化为字符串，相当于调用Date.prototype.toISOString()。然而JSON.parse()并不会解码这个字符串，需要单独把每一个字符串调用new Date()。如果直接把Date储存过于复杂，我的解决方法是储存数据前在Date前加+，将其直接变为Number来储存，这样就不会受JSON.stringify()和JSON.parse()影响，需要转化回来时再使用new Date()。

demo——jquery实现简单todos

发表于 2016-10-04   |   分类于 demo

  • demo地址：jquery-todos

  • 源码：github源码

  • 依赖：jquery

  • 特点：

  1. 纯jquery实现简单的todos，未使用其他框架。
  2. 数据存储在localStorage。
  3. 样式和结构使用之前做的vuex-todos。

  • 感想与总结：

  1. 用jquery把之前用vue写的todos重构了一编，感觉jquery也不赖。
  2. jquery具体思路这样：
    • 初始化时init(),初始化数据，绑定事件bindEvents(),运行一次render()。
    • bindEvents()中将所有的事件绑定在目标上。
    • 需要重新渲染的事件，的最后执行render()
    • 每一次render()，先empty，再根据数据重新渲染。
  3. 虽然这样也能较好的实现，但是相对vue耦合度还是太高，很多事件都绑定一个render()
  4. 不过总体来说，jquery比原生js还是方便了许多，兼容性也很强。
  5. 我觉得以后大概每学一门框架都要先写一份todos了（笑~）

demo——vue+jqueryAjax实现天气预报

发表于 2016-09-23   |   分类于 demo

  • demo地址：vue+jQueryAjax实现天气预报

  • 源码：github源码

  • 依赖：vue，jQuery，和风天气API

  • 特点：

  1. vue + jQuery的ajax实现简单的天气预报。
  2. 初始化时发送ajax获得本机ip的城市，再发送ajax返回天气预报。
  3. 城市下方的「点击切换」点击以后，可以输入国内其他城市名切换城市发送ajax。
  4. 「点击切换」输入内容以后出现浮出框供选择。
  5. css过渡虽然看似很简单，实际上发送ajax和css过渡异步进行，css过渡显示和ajax返回不堵塞。
  6. 写了一点响应式，使用flex布局，IE10+。

  • 感想与总结：

  1. 第一次用jQuery的ajax，稍微了解了一定的ajax。
  2. vue和ajax结合，途中也遇到了很多坑，最后用vue的created和transition解决了。
  3. 一共用3个ajax，流程是这样的：
    1. 初次打开时直接发送ajax返回本机ip。
    2. ip为参数发送ajax返回所在城市名。
    3. 城市名为参数发送ajax返回天气预报。
    4. 切换城市后，再回到第3步。
  4. ajax的第4步和css过渡异步进行，过程是这样：
    1. 切换城市时同时发送ajax和css过渡隐藏。
    2. 若ajax先返回，返回值赋值给newData。当css过渡隐藏结束时将newData赋给data，再css过渡显示。
    3. 当css过渡隐藏先结束，则什么都不干/(ㄒoㄒ)/~~，待到ajax返回时，返回值直接赋值给data，再css过渡显示。
    4. 这样以来无论过渡时间设置为多久，都不会出现BUG（别问我怎么知道了，调试了好久~~~）

  2. 当设定foo = new Foo()时，foo.__proto__ -> Foo.prototype。
